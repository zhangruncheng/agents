# Claude Code 子代理集合

专为 [Claude Code](https://docs.anthropic.com/en/docs/claude-code) 设计的全面专业 AI 子代理集合，通过领域专业知识增强开发工作流程。

## 概述

此仓库包含 75 个专业子代理，扩展了 Claude Code 的功能。每个子代理都是特定领域的专家，可根据上下文自动调用或在需要时显式调用。所有代理都根据任务复杂性配置了特定的 Claude 模型，以实现最佳性能和成本效益。

### 模型配置

从 Claude Code v1.0.64 开始，子代理可以指定他们应该使用的 Claude 模型。这允许基于复杂性进行成本效益的任务委托：

- **低复杂性 (Haiku)**：简单任务，如基本数据分析、文档生成和标准响应
- **中等复杂性 (Sonnet)**：开发任务、代码审查、测试和标准工程工作  
- **高复杂性 (Opus)**：关键任务，如安全审计、架构审查、事件响应和 AI/ML 工程

可用模型（使用 Claude Code v1.0.64 的简化命名）：
- `haiku` - 高速且成本效益的简单任务
- `sonnet` - 大多数开发工作的平衡性能
- `opus` - 复杂分析和关键任务的最高能力

如果没有指定模型，子代理将使用系统的默认模型。

## 代理编排模式

Claude Code 使用这些常见模式自动协调代理：

### 顺序工作流
```
用户请求 → 代理 A → 代理 B → 代理 C → 结果

示例：“构建新的 API 功能”
backend-architect → frontend-developer → test-automator → security-auditor
```

### 并行执行
```
用户请求 → 代理 A + 代理 B（同时）→ 合并结果

示例：“优化应用程序性能” 
performance-engineer + database-optimizer → 综合建议
```

### 条件分支
```
用户请求 → 分析 → 路由到适当的专家

示例：“修复这个错误”
debugger（分析）→ 路由到：backend-architect 或 frontend-developer 或 devops-troubleshooter
```

### 审查与验证
```
主要代理 → 审查代理 → 最终结果

示例：“实现支付处理”
payment-integration → security-auditor → 验证的实现
```

## 最佳实践

### 🎯 任务委托
1. **让 Claude Code 自动委托** - 主代理分析上下文并选择最优代理
2. **明确提出要求** - 包括约束条件、技术堆栈和质量要求
3. **相信代理专业能力** - 每个代理都为其领域进行了优化

### 🔄 多代理工作流
4. **从高级请求开始** - 让代理协调复杂的多步骤任务
5. **在代理之间提供上下文** - 确保代理有必要的背景信息
6. **审查集成点** - 检查不同代理的输出如何协同工作

### 🎛️ Explicit Control
7. **Use explicit invocation for specific needs** - When you want a particular expert's perspective
8. **Combine multiple agents strategically** - Different specialists can validate each other's work
9. **Request specific review patterns** - "Have security-auditor review backend-architect's API design"

### 📈 Optimization
10. **Monitor agent effectiveness** - Learn which agents work best for your use cases
11. **Iterate on complex tasks** - Use agent feedback to refine requirements
12. **Leverage agent strengths** - Match task complexity to agent capabilities

## 故障排除

### 常见问题

**代理未被自动调用：**
- 确保您的请求清晰地表示领域（例如，“性能问题” → performance-engineer）
- 对任务类型要具体（例如，“审查代码” → code-reviewer）

**意外的代理选择：**
- 提供更多关于您的技术堆栈和要求的上下文
- 如果需要特定代理，请使用显式调用

**多个代理产生相互冲突的建议：**
- 这是正常的 - 不同的专家可能有不同的优先级
- 请求澄清：“协调 security-auditor 和 performance-engineer 的建议”

**代理似乎缺乏上下文：**
- 在您的请求中提供背景信息
- 引用以前的对话或已建立的模式

### 获取帮助

如果代理未按预期工作：
1. 检查各代理文件中的代理描述
2. 在您的请求中尝试使用更具体的语言
3. 使用显式调用来测试特定代理
4. 提供更多关于您的项目和目标的上下文
